\chapter{Methodology}
\label{sec:implementation}

This section outlines the integration of the new weight update scheme proposed by \citeauthor{jiaRemovalFeedbackLoop2025} into the hardware accelerator implemented by \citeauthor{vorhaugHyperspectralImageCompression2024}. The use of this new weight update scheme, dubbed delayed weight updates, results in a compressor that is no longer compliant with the issue 2 standard. Because of this, corresponding modifications to the predictor of the decompressor must be made to verify the results. A few open-source compressor have been developed, including the verification model presented by \citeauthor{vorhaugHyperspectralImageCompression2024}. Kevin Caldwell et al. and Daniel Báscones et al. have compressors written in the programming languages C and Java respectively \footnote{As of writing, the compressors are available online in the Github repositories \textit{Kevin-Caldwell/hyperC} and \textit{Daniel-BG/CCSDS123}.}. The latter is related to the works in \cite{basconesFPGAAcceleratorRealTime2020}, whereas the first is not supported by any published works. Both compressors are compliant with the issue 2 standard, but only implement support for the sample-adaptive encoder and with limited support for decompression. This prompts the need for an open-source decompressor with support for the hybrid encoder. Before the changes to the hardware accelerator are presented, the outline for developing such a decompressor is covered.

\section{Open-source Decompressor}

As shown in \autoref{sec:compression}, the issue 2 compressor and decompressor have many similarities. This is especially true for the predictor stage, but also applies to the processing of header configuration. To shorten the development time, and for easier integration into existing systems, the verification model written by \citeauthor{vorhaugHyperspectralImageCompression2024} is used as a starting point for the implementation of the decompressor. As opposed to the verification model by \citeauthor{vorhaugHyperspectralImageCompression2024}, the decompressor is not only needed for verification purposes. To successfully integrate and verify the delayed weight updates into the on-board compressor, the decompressor must be able to operate on full scale satellite images. This means that the execution time and memory usage of the program must be kept in mind during development. Delayed weight updates (\autoref{eq:delayed-weight-updates}) are implemented as an optional feature of the tool, allowing easier verification using other issue 2 compliant tools and test data.

To analyse the execution time and memory usage of the verification model, profiling tools were used. These track the \acrshort{CPU} time and memory usage of the program during execution, and the results are usually visualised using a flamegraph. The analysis done here only aims to give a rough overview of the most time and memory consuming sections of the program, and thus only a simple analysis is made. The stippled outline of the \acrshort{hypso} image shown in \autoref{fig:lacrau-cropped} is used as the input for the verification model during profiling. The cropped image has $80$ spectral bands and a spatial size of $70\text{x}150$ samples. The compressor is configured in reduced prediction mode with narrow local sums, using the hybrid encoder and three prediction bands, giving a compression ratio of $2.47$.

The profiling results are summarized in \autoref{tab:profile}, showing the total execution time and memory usage. Results vary depending on the characteristics of the image used, algorithm configuration, and hardware used to run the program. Thus, the numbers shown in the table should only be used as a hint as to the performance of different stages of the calculation. Most notably, the prediction stage occupies $65.2\%$ of the execution time. Additionally, a significant amount of time is spent storing the intermediate results to files. This is because moving data from memory to disk is very slow. The hybrid encoder is responsible for $83.1\%$ of the heap memory allocated by the program. Note that even for such a small image, a rather large chunk of memory (1203MB) is allocated. Based on this analysis, two main approaches to improve the performance of the decompressor are used: a full rewrite of the predictor stage, and setting the storing of intermediate results as optional. This targets the two main time consumers of the program, namely the predictor and saving of outputs to disk.

Python is an interpreted language with dynamic typing. Such programming languages are known to be significantly slower than traditional compiled languages with static typing. Commonly, Python libraries such as Numpy are implemented in compiled languages such as C, C++ or Rust. By use of special libraries and builtin support in the Python language, bindings to the compiled library are created that allow classes and other objects to be used by Python. This combines the simple syntax and fast development times enabled by Python with the high speeds of precompiled libraries. In order to improve the performance of the predictor stage while reusing the encoder and header readers of the old verification model, this work rewrites the predictor stage in C++. By use of the library \textit{pybind11}, bindings are created that allow the new predictor class to be instantiated by the old Python framework. The interface of the new class is identical to that of the old class, while allowing data to be passed between the C++ and Python programs at runtime. For the rest of this document, the term \textit{verification model} will refer to the modified tool by \citeauthor{vorhaugHyperspectralImageCompression2024} including the rewritten predictor and decompression support.

\subsection{Implementation}

\begin{figure}[ht]
	\begin{center}
		\includegraphics[width=1\textwidth]{figures/verification-model.png}
	\end{center}
	\caption{Outline of compressor (left) and decompressor (right) of the verification model. The predictor, marked in red, is rewritten in C++.}
	\label{fig:verification-model}
\end{figure}

\autoref{fig:verification-model} shows the general outline of the compressor and decompressor of the verification model. The header reader is reused from the software developed by \citeauthor{vorhaugHyperspectralImageCompression2024} with one minor change. During decompression, the header reader may now use the header binary encoded in the compressed bitstream as its input. The format of this encoded header is identical to that of the header binary file used during compression, allowing the reuse of the module. The header reader may take optional tables and error limit tables as inputs when these are used. During decompression, if periodic error limit updates are used, error limit tables are populated from error limits that are encoded in the compressed bitstream. If no header binary is provided during compression the header reader will use default header configuration values. The header configuration is passed from the header reader to the predictor and encoder modules for use during the calculations.

\begin{figure}[ht]
	\begin{center}
		\includegraphics[width=0.65\textwidth]{figures/ccsds-flow-chart-compressor.png}
	\end{center}
	\caption{Flowchart for the C++ prediction loop for compression (left) and decompression (right). Samples are iterated in the \acrshort{BIP} order, with reconstruction during decompression marked in red.}
	\label{fig:flow-chart-compressor}
\end{figure}

\autoref{fig:flow-chart-compressor} shows flowcharts for the C++ prediction loops of the compressor (left) and decompressor (right). Samples are stored internally in \acrshort{BIP} ordering, and are iterated correspondingly. Calculations are done in a serial manner closely following the dependency flow shown in \autoref{fig:dependencies}, grouped by local sums and differences, prediction, quantization, sample representatives, mapping and weight updates. Each intermediate value has an associated class method that takes its dependant values as inputs and returns a single value. This avoids hidden side effects, resulting in coherent procedures that make the flow of the program easy to follow, as opposed to modifying data in place. For the special case of $t=0$, only a handful of the values need to be calculated, and other intermediaries are set to zero. During decompression, the quantizer indices, prediction residuals and image samples are reconstructed as described in \autoref{sec:predictor}.

Inspired by the compressor written in Java by Báscones et al., the C++ predictor makes use of a sampler class to allow selectively storing intermediaries. Each intermediate value has a corresponding sampler object that wraps a dynamically allocated array for storage. At the call site of the intermediate value calculation, the sampler objects sample method wraps the calculation, returning the result and storing it internally. If sampling is disabled, results are not stored and the corresponding arrays are never allocated, reducing memory requirements. The internal arrays of the sampler objects are retrieved and stored to files. This provides the same output files of the original verification model, allowing use for verification of the hardware accelerator with minimal modifications to the verification framework. Values that are needed to fulfill $z-1$ and $t-1$ dependencies, such as sample representatives and local differences, are always stored in their corresponding arrays. Doing this simplifies the implementation at the cost of larger memory requirements as opposed to using specialized buffers.

To fully support decompression, the hybrid encoder class of the old software is also modified, still using the Python language. \autoref{lst:pseudocode-decoder} shows pseudocode for the reconstruction of mapped quantizer indices encoded using the hybrid encoder. Before decoding the image body, the final accumulator values and \acrshort{V2V} active prefixes of the encoder are read from the image tail. Processing order is determined by the associated header configuration field, and the decoder iterates in reverse. If periodic error limit updates are used during the processing order \acrshort{BI}, encoded error limits must be decoded at certain intervals. For $t=0$, samples are encoded as is and may be decoded by reading an unsigned integer of size $D$ from the bitstream. Otherwise, samples are decoded as either high- or low-entropy codes based on the average estimate. The average estimate is updated after sample decoding based on the decoded sample. To allow reconstruction of rounding errors introduced during accumulator rescaling (\autoref{eq:accumulator}), the encoder encodes extra bits corresponding to the \acrshort{LSB} of the accumulator before rescaling. For full details on the decoding process see the work by \citeauthor{blanesHYBRIDENTROPYENCODER} \cite{blanesHYBRIDENTROPYENCODER}.

\begin{lstlisting}[
    float=ht,
    caption={Pseudocode for hybrid decoder.},
    label=lst:pseudocode-decoder,
    language=Python]
accumulator, active_prefixes = decode_image_tail()
counter                      = get_counter_value(t_max)

for all z and t: # iterate in reverse order
    mapped_quantizer_index[z, t] = decode_sample(z, t)
    if ordering = BI and prediodic_error_limit_updating:
        header.error_limits[z] = decode_error_limts(z)

def decode_sample(z, t): # pops bits from compressed bitstream
    if t = 0: # first sample encoded as is
        mapped_quantizer_index[z, t] = read_D_bit_integer()
    else:
        if accumulator[z, t] * 2^14 >= threshold[0] * counter[t]:
            mapped_quantizer_index[z, t] = decode_high_entropy(z, t)
        else:
            mapped_quantizer_index[z, t] = decode_low_entropy(z, t)

    t_next = t-1
    counter[t_next] = get_counter_value(t_next)

    # update accumulator
    if counter[t_next] = 2^rescaling_counter_size - 1: # rescaling
        rounding_bit = bitstream.pop()
        updated_accumulator = 2 * accumulator[z, t] - 4 * mapped_quantizer_index[z, t] - 1

        if rounding_bit = 0 and updated_accumulator_lsb = 1:
            updated_accumulator += 1 # ensure even
        else if rounding_bit = 1 and updated_accumulator_lsb = 0:
            updated_accumulator += 1 # ensure odd
        accumulator[z, t_next] = updated_accumulator

    else: # normal case
        accumulator[z, t_next] = accumulator[z, t] - 4 * mapped_quantizer_index[z, t]
\end{lstlisting}

\subsection{Verification}

Before introducing the change in the weight update scheme proposed by \citeauthor{jiaRemovalFeedbackLoop2025}, the verification model is implemented in compliance with the issue 2 standard. This allows using existing tools and data for verification. Verification of the C++ predictor and new decompression feature is done using compression and decompression tools developed by the French space agency \acrfull{CNES}, in addition to test vectors provided by the \acrshort{CCSDS} working group. Test vector \textit{Test1-20190201} is designed to provide sufficient coverage of the configurations and corner cases of the issue 2 algorithm. It contains a total of 2021 test cases, each containing a test image with accompanying header binary, optional tables and the compressed bitstream. The test vector does not include references for the decompressed images, with potential loss. To facilitate verification of the decompression feature, all test cases were decompressed using the \acrshort{CNES} decompressor to provide the necessary reference files.

Of the 2021 test cases in the test vector, only 161 use the hybrid encoder, which is the only encoder supported by the decompressor. To provide a more complete verification of the predictor of the decompressor, a setup where mapped quantizer indices were passed directly from the output of the compressor predictor to the input of the decompressor predictor was used (\autoref{fig:encoder-bypass}). This bypasses the entropy encoder, allowing the use of the full test vector, including test cases configured for use of the sample-adaptive and block-adaptive encoders.

To allow easier debugging of the new predictor module, the sample class allows setting a reference file for each intermediate value. Calculated values are then compared with the reference during runtime, notifying the user of deviating values. The old Python predictor module is used to generate reference files.

To verify and evaluate the compression performance of the delayed weight update scheme, a set of coastline images taken by the satellites \acrshort{hypso}-2, Aviris and Landsat are used. The main image set is from \acrshort{hypso}-2, consisting of ten images (\autoref{app:coastline-images}). Coastline images are chosen to allow comparison with the results of \citeauthor{jiaRemovalFeedbackLoop2025}, and the images are compressed using the same absolute error limits, \acrfull{PAE}, $\{0, 1, 3, 7, 15\}$. Visual inspection of decompressed images, in combination with \acrshort{PSNR}, is used to show that images can be reconstructed from the compressed bitstream. Additionally, images compressed and decompressed using the lossless mode are compared, during which decompressed images should be identical to the originals.

Compression performance is also evaluated across a range of prediction bands $\{0,...,12\}$ and compared to results without delayed weight updates. This provides data that can be used to determine the best number of prediction bands suited for \acrshort{hypso} images. This test, in addition to single image tests done from Aviris and Landsat, are done using relative and absolute error limits of 16 and 4 in order to observe effects of relative error limits.

\section{Hardware Accelerator}

\begin{equation}
	\omega^{(i)}_{z, \text{unclipped}}(t+1)=
	\begin{cases}
		\omega^{(i)}_z(t),                                                                                                                   & x \le 2,          \\[6pt]
		\omega^{(i)}_z(t-3) + \omega^{(i)}_{z, \text{offset}}(t-3),                                                                          & x = 3,            \\[6pt]
		\left\lfloor \dfrac{1}{2}\left(\omega^{(i)}_z(t-3) + \omega^{(i)}_{z, \text{offset}}(t-3) + \omega^{(i)}_z(t) \right) \right\rfloor, & \text{otherwise}.
	\end{cases}
	\label{eq:delayed-weight-updates-modified}
\end{equation}

The hardware accelerator developed by \citeauthor{vorhaugHyperspectralImageCompression2024} is updated in accordance with the delayed weight update scheme proposed by \citeauthor{jiaRemovalFeedbackLoop2025}, presented in \autoref{eq:delayed-weight-updates}. By use of this new approach, the previously indivisible weight update calculation of pipeline stage three (\autoref{fig:vorhaug-pipeline}) can be split into four distinct stages as shown in \autoref{fig:new-stage3}. The prediction calculation is done in parallel with the weight update, with stage 3a and 3b calculating $\hat{d}_z(t)$ and $\breve{s}_z(t)$. $\tilde{s}_z(t)$ and $\hat{s}_z(t)$ are calculated in stage 3c before $\Delta_z(t)$ and $m_z(t)$ are found in stage 3d. As described by \citeauthor{sanchezReducingDataDependencies2022}, speculative execution of the two possible outcomes of the weight update is performed in two parallel execution paths. Once the required values to find the sign of the prediction error (\autoref{eq:math-opt}) resolves in stage 3d, the correct value is chosen.

The hardware presented in \autoref{fig:weights-hw} is responsible for selecting the correct, updated weight value and is duplicated for each element $i$ of the weight vector (\autoref{eq:weight-vector}). A new weight value is selected every cycle in stage 3a, allowing the calculation of $\hat{d}_z(t)$ to proceed in the following cycle. The module remains the same as that of \citeauthor{vorhaugHyperspectralImageCompression2024} (\autoref{fig:vorhaug-weights-hw}) except for one additional multiplexor. This allows the same weight to be reused for the first three samples of each line (\autoref{eq:delayed-weight-updates}). If $x\le3$ the output of the weight update pipeline, $\omega_{z,updated}^{(i)}(t)$, is selected.

	\begin{center}
		\includegraphics[width=1\textwidth]{figures/ccsds-stage3-hw-pipeline.png}
	\end{center}
	\caption{Hardware of the weight update pipeline. For $x>3$ weight refinement is applied before the correct speculative path is selected.}
	\label{fig:weights-pipeline-hw}
\end{figure}

\autoref{fig:weights-pipeline-hw} shows the pipelined calculation of the weight updates. Stages 3a to 3c calculate the weight offset (\autoref{eq:weight-offset}) and adds the weight of $t-3$. Stage 3d applies weight refinement for $x>3$, before the speculative paths started in stage 3b are resolved based on the sign of the prediction error. Finally, the updated weight is clipped within the bounds of $\omega_{min}$ and $\omega_{max}$ and made available to the hardware of \autoref{fig:weights-hw}. Ideally, the operations of the weight update calculation should be split between pipeline stages such that the path delay is equal in the stages. However, as long as neither of the weight update stages constitute the critical path of the design the selected distribution is good enough.

\begin{figure}[ht]
	\begin{center}
		\includegraphics[width=0.55\textwidth]{figures/ccsds-new-stageD9.png}
	\end{center}
	\caption{Sample representative stage split into three stages D+9a, D+9b, and D+9c, reducing the new critical path after splitting the weight update stage.}
	\label{fig:new-stageD9}
\end{figure}

Timing analysis after splitting the weight update stage showed that the calculation of the sample representatives in stage D+9 was the new critical path. To further increase the maximum achievable clock speed, stage 9 is split into three stages as shown in \autoref{fig:new-stageD9}. Firstly, calculation of the clipped quantizer bin center $s'_z(t)$ (\autoref{eq:quantizer-bin-center}) is split into two stages. Here, $s'_{z,1}(t) = q_z(t)(2 m_z(t) + 1)$ and is calculated in parallel with the mapped quantizer index $\delta_z(t)$ in stage D+9a. $s'_{z,2}(t)$ in stage D+9b adds the predicted sample value $\hat{s}_z(t)$ and clips the result within $s_{min}$ and $s_{max}$ giving the final value of $s'_z(t)$. The double-resolution sample representative $\tilde{s}''_z(t)$ and sample representative $s''_z(t)$ are calculated in stage D+9c.

The full, updated pipeline is presented in \autoref{fig:new-pipeline}. To facilitate the increased number of pipeline stages, the data shift registers responsible for providing data when needed are also updated. This is simply a matter of updating the length of the registers and adjusting the taps for data retrieval.

\subsection{Verification}

Verification of the changes in the hardware accelerator is done reusing the verification framework developed by \citeauthor{vorhaugHyperspectralImageCompression2024}, with only slight modifications. Here, the test data is applied to the simulated design and to a golden model. Outputs are then compared to determine the correctness of the design. The verification framework is written using Python, with support for using different \acrshort{RTL} simulators such as Questa and GHDL. The golden model is the updated verification model. Modifications are made to read outputs from the golden model from files rather then accessing them directly from the old Python predictor module. Checks of the intermediate values are also updated in accordance to the updated predictor pipeline.

Randomly generated test data is used as the inputs of the verification framework. For each test, a random header configuration selected from the supported values (\autoref{tab:compliance}) and a random image are generated. Random images are generated based on three different methods: uniform distribution, normal distribution and singular values. A method is selected at random and image samples are selected from the corresponding distribution. For the singular value method, all samples have the same value. The value is selected as either the sample minimum, middle or maximum sample value. To ensure sufficient verification coverage of different combinations of feature parameters and corner cases, coverage data is collected (\autoref{app:verification-coverage}). The data shows how many of the 512 test cases uses each of the available parameters.

\subsection{Performance Evaluation}

The performance of the updated accelerator is done in terms of hardware resource utilization, power usage, and maximum clock frequency. Since the accelerator processes one sample per clock cycle, after the pipeline is filled, the clock speed can be directly translated to sample throughput. The design is implemented for two different \acrshort{FPGA}s: Zynq-7030 and Zynq XCZU7EV. The former is used on-board the \acrshort{hypso} satellites, and the latter is used by \citeauthor{jiaRemovalFeedbackLoop2025} in their work presenting the delayed weight update scheme. Maximum clock frequency is estimated using static timing analysis tools.

The design is evaluated for the full accelerator and for the predictor module only. This allows comparison with implementations in the literature that only implement the predictor. The header configuration used during evaluation is based on the work by \citeauthor{blanesPerformanceImpactParameter2019} and is presented in \autoref{tab:parameters}. These values are also the same as those used by \citeauthor{vorhaugHyperspectralImageCompression2024}.

\subsection{On-board Testing}

To verify the accelrator on real hardware, the design is integrated into the \acrfull{OPU} of the \acrshort{hypso}-2 satellite. The central component of the \acrshort{OPU} is the Zynq-7030 \acrshort{MPSoC} from AMD (formerly Xilinx). The chip contains dual ARM \acrshort{CPU}s and an integrated Kintex-7 \acrshort{FPGA}. \autoref{fig:OPU} shows an overview of the \acrshort{OPU}. Images are captured by the \acrfull{HSI} and passed to the software running on the ARM \acrshort{CPU}s for preprocessing. Raw image cubes are then stored in \acrshort{RAM} and retrieved by the cubeDMA (Direct Memory Access) module inside the \acrshort{FPGA} fabric for compression using the \acrshort{CCSDS} compressor. The compressed image is moved back to \acrshort{RAM} and then stored to an SD-card. Once a downlink is scheduled, the image is retrieved and passed to the separate \acrfull{PC} and downlinked using S-band radio. For more details on the \acrshort{hypso}-1 and \acrshort{hypso}-2 \acrshort{OPU}, please refer to the work by \citeauthor{langerRobustReconfigurableOnBoard2023} \cite{langerRobustReconfigurableOnBoard2023}.
